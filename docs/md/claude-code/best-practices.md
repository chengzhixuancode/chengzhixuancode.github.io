## 简介

Claude Code 是 Anthropic 推出的 CLI 工具，让 Claude 3/4 系列大模型直接在你的本地开发环境中协助开发。本手册总结了社区验证过的最佳实践，帮助你最大化提升开发效率。

---

## 基础使用原则

### 1. 你是负责人，Claude 是协作者

```
✓ 你对项目质量和安全性负责
✓ Claude 提供建议和执行重复劳动
✓ 始终审查 Claude 写的每一行代码
✓ 不要接受你不理解的代码
```

### 2. 小步快跑，快速迭代

- 一个会话解决一个问题
- 大任务拆成小步骤
- 每次变更后及时验证
- 发现问题立即调整

### 3. 保持上下文清晰

- 无关问题开新会话
- 使用 `/clear` 清理冗余上下文
- 大项目拆分多个会话处理

## 高效提问技巧

### ✅ 好的提问模板

**Bug 排查：**
```
在 src/auth/login.ts 第 45 行，当用户输入错误密码时，期望显示"密码错误"提示，但实际页面直接崩溃了。错误信息是：Cannot read property 'message' of undefined。帮我找出原因并修复。
```

**功能开发：**
```
在这个 Next.js 项目中，帮我添加一个用户列表分页组件。
要求：
- 使用现有的 Tailwind CSS 样式
- 支持每页 10/20/50 条选项
- 点击页码切换页面，触发 onPageChange 回调
- 遵循项目现有的 TypeScript 类型定义
```

**代码重构：**
```
请帮我重构这个函数，目标：
- 拆分多个小函数提高可读性
- 添加适当的类型注解
- 保持原有功能不变
- 遵循项目现有的代码风格
```

### ❌ 不好的提问方式

| 不好 | 为什么不好 |
|------|-----------|
| "帮我写个登录功能" | 缺少技术栈、项目上下文、具体要求 |
| "这里有问题，帮我看看" | 没有说明问题在哪，期望是什么 |
| "优化一下这个项目" | 范围太大，没有明确优化目标 |

### 🎯 提问三要素

1. **在哪里** → 文件路径、行号
2. **发生了什么** → 问题现象或需求描述
3. **期望是什么** → 期望结果是什么

## 开发工作流

### 推荐的功能开发流程

```mermaid
graph LR
A[需求理解] --> B[探索代码库]
B --> C[制定实现计划]
C --> D[获批准入]
D --> E[逐步实现]
E --> F[代码质量检查]
F --> G[测试验证]
G --> H[提交]
```

**详细步骤：**

1. **探索理解阶段**
   - 让 Claude 先探索代码库：`Agent 子agent_type=Explore "帮我找一下项目中处理用户认证的所有文件"`
   - 阅读关键文件，理解现有架构和代码风格

2. **规划阶段**
   - 使用 `EnterPlanMode` 制定实现计划
   - 讨论不同方案的利弊
   - 获得你的确认后再开始编码

3. **实现阶段**
   - 分模块实现，一个功能一个功能来
   - 每次实现后及时测试
   - 遇到问题即时调整

4. **检查阶段**
   - 使用 `check` 代理检查代码质量
   - 验证是否符合项目规范
   - 检查潜在的安全问题

5. **提交**
   - 使用 `/commit` 生成规范的提交信息

### 调试工作流

1. **提供完整信息** → 错误信息、堆栈、复现步骤
2. **定位相关代码** → 让 Claude 读取相关文件
3. **分析假设** → 讨论可能的原因
4. **逐一验证** → 逐个排除，找到根因
5. **修复验证** → 修复后验证问题解决

### 重构工作流

1. **理解现有功能** → 先理解要重构的代码做了什么
2. **明确重构目标** → 性能/可读性/架构调整
3. **保持接口不变** → 不改变外部行为
4. **测试验证** → 确保原有功能正常

## 工具使用策略

### 善用子代理

| 代理类型 | 适用场景 | 何时使用 |
|---------|---------|---------|
| **Explore** | 代码库探索 | 需要快速找到相关代码时 |
| **Plan** | 架构设计 | 需要制定实现计划时 |
| **implement** | 功能实现 | 按计划实现功能 |
| **debug** | 问题调试 | 排查疑难杂症 |
| **check** | 质量检查 | 实现完成后检查代码 |
| **research** | 技术调研 | 需要查找技术方案 |

**最佳实践：**
- 让专业代理做专业的事
- 多任务可以并行启动多个代理
- 主会话保持思路清晰，让子代理做苦力

### 并行执行多个工具

```typescript
// 好的做法：一次调用多个工具并行处理
// 同时读取多个相关文件，节省时间
Read 文件1
Read 文件2
Read 文件3

// 不好的做法：一个个顺序读取
```

### 使用任务管理

- `TaskCreate` - 创建任务跟踪进度
- `TaskUpdate` - 更新任务状态
- `TaskList` - 查看整体进度

对于复杂多步骤任务，任务管理能帮你和 Claude 都保持清晰。

## 项目配置最佳实践

### 1. CLAUDE.md 文件

**作用：** 让 Claude 快速了解你的项目

**推荐内容：**
```markdown
# 项目名称

## 技术栈
- 前端：React 18 + TypeScript + Tailwind CSS
- 后端：Node.js + Express
- 数据库：PostgreSQL

## 项目结构
src/
  components/    # 通用组件
  pages/         # 页面组件
  api/           # API 接口
  utils/         # 工具函数

## 代码规范
- 使用 TypeScript，尽量避免 any
- 组件使用函数式写法
- 遵循 ESLint 配置
- 提交信息遵循 Conventional Commits

## 常用命令
- npm install 安装依赖
- npm run dev 启动开发服务器
- npm run build 构建
- npm test 运行测试
```

**生成方式：** 运行 `/init` 自动生成

### 2. 权限配置

**原则：最小权限**

```json
// .claude/settings.json（项目共享）
{
  "permissions": {
    "allow": [
      "Read(*)",
      "Glob(*)",
      "Grep(*)",
      "Bash(npm install:*)",
      "Bash(npm run build:*)",
      "Bash(npm test:*)"
    ]
  }
}
```

```json
// .claude/settings.local.json（个人，不要提交）
{
  "permissions": {
    "allow": [
      // 个人允许的额外权限
      "Bash(git:*)"
    ]
  }
}
```

**最佳实践：**
- 项目级配置放到 `settings.json` 提交到仓库共享
- 个人个性化配置放到 `settings.local.json`，在 `.gitignore` 中忽略
- 危险操作（如 `git push -f`、`rm -rf`）保持手动确认

### 3. Hooks 自动化

**预执行钩子：** 命令运行前自动检查
```json
{
  "hooks": {
    "preCommand": [
      "npm run lint -- --fix {{filename}}"
    ]
  }
}
```

**后执行钩子：** 命令完成后自动处理
```json
{
  "hooks": {
    "postCommand": [
      "git status",
      "echo 'Command completed at $(date)'"
    ]
  }
}
```

**常用场景：**
- 自动格式化代码
- 自动运行 lint 检查
- 自动运行相关测试

### 4. 自定义技能

为项目特定任务创建自定义 slash 命令，比如：
- `/before-frontend-dev` - 读取前端开发指南
- `/before-backend-dev` - 读取后端开发指南
- `/check-frontend` - 检查前端代码规范
- `/finish-work` - 提交前检查清单

### 5. .gitignore 配置

确保忽略 Claude 生成的本地配置文件：

```gitignore
# Claude local configuration
.claude/settings.local.json
.claude/tmp/
.claude/sessions/
```

项目共享配置（`settings.json`）可以提交，个人本地配置不要提交。

## 高级技巧

### 1. Token 成本控制

Claude Code 按 token 收费，合理控制成本：

```
✓ 小问题开小会话，及时 /clear 清理
✓ 使用 compact 压缩上下文
✓ 只让 Claude 读取需要的文件，不是整个项目
✓ 复杂任务拆分多个会话
✓ 使用 /insights 查看 token 使用情况
```

**估算技巧：**
- 1K token ≈ 750 中文汉字
- Claude 4 比 Claude 3.5 Sonnet 贵很多，简单任务用 Sonnet
- 只有复杂推理任务才需要用 Claude 4 Opus

### 2. 模型选择策略

| 模型 | 适用场景 | 相对成本 |
|------|---------|----------|
| Claude 3.5 Sonnet | 日常编码、bug 修复 | 💰 较低 |
| Claude 4 Sonnet | 复杂设计、大型重构 | 💰💰 中等 |
| Claude 4 Opus | 架构设计、深度推理 | 💰💰💰 较高 |

根据任务复杂度选择合适的模型，不用大材小用。

### 3. 循环检查

```
/loop 10m npm test
```
定期运行测试，适合调试间歇性失败的测试

### 2. 状态行集成

运行 `/statusline` 配置美观的状态行，实时显示：
- Git 分支
- 项目状态
- 关键信息

### 3. 大规模批量处理

使用 `/batch` 处理多个分支上的相同变更，比如批量修复安全漏洞。

### 4. 内存系统

Claude Code 支持持久化内存，在 `.claude/projects/xxx/memory/` 中：

- `user.md` - 你的角色和偏好
- `feedback.md` - 团队的编码规范偏好
- `project.md` - 项目背景和目标
- `reference.md` - 有用的资源链接

维护好内存能让 Claude 更贴合你的团队习惯。

### 5. 利用 Fast Mode

- 使用 `/fast` 切换到快速模式，响应更快
- 复杂推理任务切回普通模式
- 根据任务类型灵活切换

## 常见误区

### ❌ 误区 1：一次性扔给 Claude 太大的任务

**问题：** 上下文溢出，质量下降
**解决：** 拆分，小步快跑

### ❌ 误区 2：让 Claude 猜你的需求

**问题：** 结果不符合预期，反复返工
**解决：** 给够上下文，明确具体要求

### ❌ 误区 3：不审查直接使用 Claude 写的代码

**问题：** 引入 bug 或安全漏洞
**解决：** 永远审查每一行代码，理解后再合并

### ❌ 误区 4：在一个会话处理多个不相关问题

**问题：** 上下文混乱，干扰模型判断
**解决：** 不同问题开不同会话

### ❌ 误区 5：过度抽象，提前优化

**问题：** 为不存在的需求创建复杂抽象
**解决：** YAGNI 原则（You Ain't Gonna Need It），只解决当前问题

## 故障排查

### 问题 1：Claude 找不到文件

**解决：**
- 检查路径是否正确，使用绝对路径
- 运行 `Bash pwd` 确认当前工作目录
- 使用 `Glob` 模式搜索文件

### 问题 2：上下文太长，响应变慢

**解决：**
- 使用 `/clear` 清空上下文
- 开新会话
- 把大任务拆分

### 问题 3：权限不够

**解决：**
- 使用 `/update-config` 更新权限配置
- 检查 `settings.json` 和 `settings.local.json`
- 添加对应的权限规则

### 问题 4：工具调用失败

**解决：**
- 检查网络连接
- 检查 API 密钥是否有效
- 运行 `/debug` 查看详细错误日志

### 问题 5：结果质量下降

**解决：**
- 清理上下文，开新会话
- 更清晰地描述需求
- 拆分问题，分步解决
- 尝试使用不同的提示方式
